# 单轮对话验收标准与停止条件的可见性

## 来源与借鉴理由

**Manus system prompt**（leaked-system-prompts）在 Agent 停止机制上有一个明确设计：
Agent 在完成一批工作后会输出一段「已完成工作的总结」和「下一步建议」，
让用户明确感知到「这一轮做了什么、做完了、下一步是什么」。

**OpenPawz** 的 Agent 面板在每一轮结束时展示一个清晰的「会话摘要」区域，
用户可以看到本轮 AI 执行了哪些操作、哪些成功、哪些跳过。

这两者共同指向一个缺失：**用户不应该只能通过观察工具调用卡来推断「AI 到底做了什么」。**

## 当前差距

### 停止机制本身是合理的

当前系统的三层停止机制是设计合理的：
1. LLM 自然停止（不再输出 tool_call）
2. max_tool_rounds = 12 兜底
3. ask_user 中断

这三层组合覆盖了正常、超限、需要输入三种情况，没有明显的逻辑缺陷。

### 问题在于「完成感知」而非「完成逻辑」

**用户看到的东西**：
- 一系列工具调用卡（ToolCallCard），每个显示工具名 + 参数摘要 + 状态
- 流结束后出现一段 AI 的文字回复
- 没有「本轮完成了什么」的结构化总结

**用户无法感知的东西**：
- 这一轮创建/修改了哪些资产（工具调用卡展开才能看到，不展开完全不知道）
- 一致性检查的结果是什么（经过了 check_consistency，但结果被折叠在工具卡里）
- 本轮是否完整完成了用户的请求（还是做了一部分，期待下一轮继续）
- Director 是否认为「自己完成了」，还是「暂停了等待用户确认」

### 「验收标准」在哪里定义

当前系统对「一轮对话的完成」**没有显式的验收标准**，完全依赖 LLM 自行决定何时停止输出工具调用。
system.txt 中没有任何「你完成了什么条件就可以结束」的指令，
也没有「结束时应该输出什么样的总结」的约束。

这导致两个问题：
1. **过早停止**：LLM 在完成部分工作后认为「够了」就停，用户实际需要的内容没做完
2. **没有明确的完成信号**：用户看到文字停了，不确定 AI 是「做完了」还是「等我指令」

## 适合性判断

**适合**。这是一个纯粹的「完成感知」问题，不需要改变任何底层架构，
只需要在 prompt 层和/或 UI 层增加「完成总结」机制。

## 对创作控制感的影响

**强烈改善**。用户在每轮结束时能清楚知道「AI 这一轮做了什么、做了多少、做完了吗」，
是创作控制感的核心要素之一。

## 对 workbench 协同的影响

改善「右栏 Agent 面板 ↔ 中栏编辑器」的协同。
当用户知道本轮创建/修改了哪些资产（通过完成总结），
他可以直接在左栏点击跳转，而不是茫然地猜测「AI 改了哪里」。

## 对 1.0 用户价值的影响

**重要但不阻塞 1.0**。当前体验是「可用但不清晰」，用户可以工作，但认知负担高。
如果 1.0 前有时间优化，优先级较高；否则可以在 1.1 阶段补充。

## 建议修改范围

### 方案 A（推荐）：Prompt 层约束完成总结格式

在 `apps/backend/app/prompts/director/system.txt` 增加一段「完成回复规范」：

```
### 完成回复规范

每轮对话结束时，在文字回复中必须包含以下结构（紧凑即可，不需要冗长）：

**已完成：**
- [动作] [资产名/文件名]（如：创建了 NPC「白狐」）
- [动作] [资产名/文件名]（如：修改了场景「幽暗森林」的背景描述）

**注意事项：**（仅有一致性警告或错误时才出现）
- [问题描述]

若本轮没有执行任何写入操作，不需要输出「已完成」段落。
```

优点：实现成本极低（只改 prompt），前端完全不需要改动。
缺点：是 AI 自己写的文字，可能格式不稳定；不能在 UI 上做结构化展示。

### 方案 B：前端增加「本轮操作摘要」气泡

在 `done` 事件处理后，从本轮 `tool_call_result` 事件中提取所有 `workspace_mutating: true`
的操作，在 AssistantMessage 底部渲染一个「本轮已修改」徽章列表（资产名 + 操作类型）。

优点：结构化、可点击（点击跳转到资产）、样式可控。
缺点：实现成本中等，需要改 `AgentPanel.tsx` 和可能的新组件。

### 方案 C：Prompt + 前端联动（完整解法）

Prompt 层约束 Director 在完成时调用一个 `report_completion(summary)` 工具，
前端接收这个工具调用的结果，以结构化卡片渲染「完成报告」。

优点：最清晰，格式完全可控。
缺点：实现成本最高，需要新工具定义、SSE 处理、前端组件。

**建议先做方案 A（低成本验证效果），再评估是否需要方案 B。**

## 建议落地方式

- [ ] **直接改代码（方案 A）**：在 `apps/backend/app/prompts/director/system.txt`
  增加「完成回复规范」段落（约 10 行）
- [ ] **plan（方案 B/C）**：如果方案 A 效果不够，考虑进入下一个 milestone
  实现前端结构化的「本轮操作摘要」组件

## 不做的理由（完整方案 C 暂缓）

方案 C（`report_completion` 工具）引入了一个新的「无实际功能的汇报工具」，
这会给 LLM 带来额外的 token 开销，且容易被 LLM 在不该调用时调用或忘记调用。
1.0 前不引入，先用 prompt 层的文字约束验证效果。
