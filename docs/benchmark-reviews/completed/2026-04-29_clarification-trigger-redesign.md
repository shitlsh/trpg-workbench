# ask_user 触发失败的根本原因与修复方向

## 来源与借鉴理由

工业级产品（Manus、Claude Projects、Cursor Agent 模式）的 system prompt
在「何时澄清、何时直接推断」这个问题上有一致的结构化处理方式：
**先给 AI 一个宽松的「默认推断」路径，再在极少数情况下设置严格触发门槛。**

当前 system.txt 的写法恰好相反：正向触发条件非常严格
（关键分叉 + 上下文无法推断 + 已先做读取），而禁止场景也非常宽泛
（包括「可以通过合理假设后在回复中说明」这条兜底）。
结果是 LLM 总是走兜底路径——自行假设，而不触发 ask_user。

## 当前差距

### 触发规则的结构性问题

`apps/backend/app/prompts/director/system.txt` 第 80-91 行：

```
调用 ask_user 时必须满足所有以下条件：
1. 任务方向存在关键分叉，且无法从对话历史或工作空间推断
2. 已完成必要的读取工具调用
3. 每次最多 2 个问题

禁止：
- 仅为礼貌确认
- 对话历史中已有足够信息
- 规则集或工作空间配置已能决定方向
- 问题可以通过合理假设后在回复中说明  ← 这条是最大的问题
```

**问题所在**：第 4 条禁止条件「可以通过合理假设后在回复中说明」实际上覆盖了绝大多数创作场景。
LLM 在创作任务中几乎永远可以「合理假设」——因此它永远走这条逃生通道，
把假设写进回复里，而不是停下来问。

### 缺少「用户信息量不足」的专门识别路径

当前 system.txt 没有任何指令教 Director 识别「用户输入的信息量是否足够完成一次有意义的创作」。
这和「有关键分叉」是两个不同的问题：
- 「关键分叉」= 方向已知，但不确定用户偏好哪个
- 「信息量不足」= 任务描述本身就太模糊，无法确定方向

比如「帮我创建一个 NPC」——这不是「分叉」，而是「信息量根本不够」。
当前 prompt 没有给这个场景设计合适的触发路径。

## 适合性判断

**高度适合**。这是直接影响「是否能生成有意义内容」的前置问题。
对 TRPG 创作场景尤为关键：NPC/场景/线索的创建需要最少的维度信息（名字、类型、背景）
才能产出有价值的内容，而用户往往只给一句话。

## 对创作控制感的影响

**强烈改善**。当前 Director 自行假设后再生成，用户拿到的是一个「已经做好决定」的结果，
修改成本高。如果 Director 先问，用户参与了创作方向的决策，结果更符合预期，
修改需求也显著减少。

## 对 workbench 协同的影响

改善「右栏 Agent 面板 ↔ 用户」的交互节点。
当前：用户发消息 → AI 直接执行 → 用户看结果（黑箱）
改善后：用户发消息 → AI 发现信息不足 → QuestionCard 弹出 → 用户点选 → AI 基于答复执行

QuestionCard 组件已经完整实现（`QuestionCard.tsx`），数据流也通（`AgentQuestionInterrupt` → SSE），
问题只在 prompt 的触发规则设计上。

## 对 1.0 用户价值的影响

**1.0 前必须解决**。当前 ask_user 机制「有但无效」是比「没有」更糟糕的状态——
用户看到 QuestionCard 的设计但永远不触发，会形成「功能有问题」的印象。

## 建议修改范围

### 核心改动：重写 system.txt 的 ask_user 规则段

将触发逻辑从「关键分叉型」扩展为两类：

**类型 A：信息量不足型（新增）**
```
用户请求的信息量低于创作最低门槛时必须调用 ask_user：
- 只提供了资产类型，没有提供任何内容方向（如「帮我创建一个 NPC」）
- 只提供了动作，没有提供对象（如「帮我扩展一下」而没有指定资产）
- 描述模糊到任何合理假设都会导致结果方向完全不同
此时 ask_user 的问题应聚焦于收集最少必要信息，而非让用户「选方向」。
```

**类型 B：关键分叉型（现有，收紧兜底）**
```
任务方向明确但存在影响结果的关键分叉时可调用 ask_user。
禁止：仅为礼貌确认，已有足够信息可推断。
注意：「可以通过合理假设」不是不问的理由——如果假设错误会导致大量无效工作，
      应该先问。
```

**删除或修改**：删除「问题可以通过合理假设后在回复中说明」这条兜底禁止规则，
改为：「若假设错误的修改成本较低（如短文本的措辞），可以假设后在回复中说明；
若假设错误会导致整体方向偏差（资产类型、核心背景、阵营），应先问。」

### 可选改动：增加「最低信息量清单」

在 system.txt 中为常见资产类型定义最低必要信息字段：

```
资产创建最低信息门槛：
- NPC：名字 + 阵营（或关系定位） + 至少一个背景元素
- 场景：地点名称 + 时代/风格背景
- 线索：与哪个事件/角色关联 + 揭示内容方向
- 怪物：战斗角色定位（对抗型/遭遇型/Boss）+ 外形风格
若任何必要字段缺失且无法从工作空间上下文推断，调用 ask_user 收集。
```

## 建议落地方式

- [ ] **直接改代码**：修改 `apps/backend/app/prompts/director/system.txt` 的 ask_user 规则段（约 12 行）
  - 拆分为「信息量不足型」和「关键分叉型」两类触发规则
  - 删除「可以通过合理假设后在回复中说明」的兜底禁止规则，改为更精确的判断标准
  - 可选：增加常见资产类型的最低信息量清单
- [ ] **写 skill**：考虑将 ask_user 调用规范提取为独立的 `director-clarification-rules` skill，
  方便单独迭代，不和 system.txt 其他内容混在一起

## 验证方式

改动后测试以下场景：
1. 「帮我创建一个 NPC」→ 应触发 ask_user（信息量不足）
2. 「帮我给白狐加一段关于背叛的经历」→ 不应触发（信息足够，直接执行）
3. 「创建一个和主角有关的地点」→ 应触发（关联不明确）
4. 「优化一下这个 NPC 的描述」→ 不应触发（有已打开资产作为上下文）
